These notes provide a comprehensive overview of the Project Lombok tutorial by Code Decode, covering its purpose, core annotations, installation, and internal mechanics.

---

### **1. Introduction to Project Lombok**
Project Lombok is a popular Java library used to minimize "boilerplate" code. Boilerplate code refers to sections of code that must be included in many places with little to no alteration (e.g., Getters, Setters, Constructors).

*   **Problem:** In standard Java, a simple POJO (Plain Old Java Object) with 5 fields can require 50+ lines of code for getters, setters, `equals()`, `hashCode()`, and `toString()`.
*   **Solution:** Lombok replaces these manual blocks of code with simple annotations, making the codebase cleaner and easier to maintain.

---

### **2. How Lombok Works**
Lombok does not work at runtime; it works during the **compilation** phase.
1.  When you compile your code, Lombok hooks into the compilation process.
2.  It parses the annotations and injects the corresponding Java bytecode directly into the `.class` file.
3.  The source code remains clean (only annotations), but the generated bytecode contains the full implementation of methods.

---

### **3. Setting Up Lombok**

#### **Step 1: Maven Dependency**
To use Lombok in a Spring Boot project, add the following dependency to your `pom.xml`:
*   **GroupId:** `org.projectlombok`
*   **ArtifactId:** `lombok`
*   **Scope:** `provided` (required only for compilation, not needed at runtime as the code is already injected into bytecode).

#### **Step 2: IDE Configuration**
Because Lombok modifies code during compilation, your IDE (IntelliJ or Eclipse) needs to understand it to provide autocomplete features.
*   **IntelliJ IDEA:** Install the "Lombok" plugin and go to *Settings > Build, Execution, Deployment > Compiler > Annotation Processors* and check **"Enable annotation processing."**
*   **Eclipse:** You must run the Lombok `.jar` file and point it to your Eclipse installation to patch the IDE.

### **4. Core Lombok Annotations**

#### **@Getter and @Setter**
*   Generates getters and setters for all fields if placed at the class level.
*   Can be placed on individual fields if you only want it for specific variables.

#### **@ToString**
*   Generates an implementation of the `toString()` method.
*   By default, it prints the class name and each field in a name-value pair format.

#### **@EqualsAndHashCode**
*   Generates `equals(Object other)` and `hashCode()` methods.
*   It follows the standard contract: if two objects are equal, they must have the same hash code.

#### **@NoArgsConstructor**
*   Generates a private or public constructor with no parameters. Useful for JPA/Hibernate entities which require a default constructor.

#### **@AllArgsConstructor**
*   Generates a constructor with one parameter for every field in the class.

#### **@RequiredArgsConstructor**
*   Generates a constructor for all fields that are marked as `final` or `@NonNull`. This is highly recommended for **Constructor Injection** in Spring Boot.

#### **@Data**
*   This is a "shortcut" or "bundle" annotation. It combines:
    *   `@Getter` / `@Setter`
    *   `@ToString`
    *   `@EqualsAndHashCode`
    *   `@RequiredArgsConstructor`

#### **@Builder**
*   Implements the **Builder Pattern**. It allows you to create objects using a fluent API:
    *   `Student.builder().name("John").age(20).build();`
*   This is much more readable than using a constructor with many parameters.

#### **@Slf4j / @Log**
*   Automatically creates a logger field: `private static final org.slf4j.Logger log = org.slf4j.LoggerFactory.getLogger(ClassName.class);`
*   Allows you to immediately use `log.info("message")` without manual declaration.

### **5. Comparison: Before vs. After**

**Without Lombok:**
```java
public class User {
    private Long id;
    private String name;

    public User() {}
    public User(Long id, String name) {
        this.id = id;
        this.name = name;
    }
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    // equals, hashCode, and toString methods...
}
```

**With Lombok:**
```java
@Data
@NoArgsConstructor
@AllArgsConstructor
public class User {
    private Long id;
    private String name;
}
```

### **6. Key Advantages**
1.  **Readability:** The business logic isn't buried under hundreds of lines of boilerplate.
2.  **Maintenance:** If you add a new field, you don’t need to manually update the constructor, getters, setters, or `toString()` method.
3.  **Consistency:** Reduces human error (e.g., forgetting to update `hashCode` when adding a new field).

### **7. Potential Disadvantages**
1.  **IDE Dependency:** Every developer on the team must have the Lombok plugin installed in their IDE.
2.  **Hidden Logic:** Since the code is generated at compile time, it can sometimes be harder to debug if you aren't familiar with how Lombok generates methods (though this is rarely an issue).
3.  **Library Dependency:** Your project becomes dependent on a third-party library for basic Java functionality.
